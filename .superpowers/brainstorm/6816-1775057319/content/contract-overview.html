Test Contract System — Overview

Every system gets a contract defining what MUST be tested. No code ships without satisfying its contract.

How It Works

1

Define Contracts

For each system, list the behaviors and invariants that MUST hold. These become the game's living spec.

2

Test Against Contracts

Each contract item maps to one or more test cases. Gaps are visible. Coverage is measured against the contract, not lines of code.

3

Enforce on Every Change

New features must add contract items AND tests. AI cannot deliver code without proving it satisfies the contract.

Three Test Layers

Layer 1: Unit Contracts

Each engine system (Combat, Inventory, Quest, Dialog, Save, RNG, Storyline) has a contract listing every behavior that must be verified in isolation. Example: "CombatSystem: dead actors cannot take turns, critical hits deal 2x, status effects tick down each turn"

Layer 2: Integration Contracts

Cross-system behaviors that must work together. Example: "Winning a battle grants EXP → triggers level-up → updates stats → save/load preserves new level"

Layer 3: Player Experience Contracts

End-to-end scenarios verifying the actual player journey. Example: "Player walks to Elder → talks → accepts quest → fights goblin → wins → quest completes → can save and reload"
